## Project Goal

Build a production-minded Flash Sale application using:

- Backend: .NET 8, C#, ASP.NET Core Web API
- Frontend: React, TypeScript
- Database: PostgreSQL
- Cache/session/reservation support: Redis
- Optional async processing: RabbitMQ, Kafka, or cloud-native queue later
- Packaging: Docker and Docker Compose first
- Scale target: high-concurrency flash sale traffic
- Deployment target: Azure or AWS

The app should start as a practical modular monolith, then evolve toward scalable deployment patterns when the core behavior is stable.

## Product Scope

The application will support:

- Customer registration and login
- Product listing and product detail
- Flash sale campaigns with start time, end time, sale price, and limited quantity
- Customer purchase flow during an active sale
- Inventory protection to avoid overselling
- Temporary reservation/hold flow with expiry
- Order creation
- Admin product management
- Admin campaign management
- Admin order monitoring
- Realtime or near-realtime sale status later

## Core Engineering Problems

The most important technical risks to solve are:

- Prevent overselling when many users buy at the same time
- Prevent duplicate orders when a user retries or double-clicks
- Keep the API stateless so it can scale horizontally
- Use Redis for hot campaign data, reservation state, rate limiting, or queue/waiting-room support
- Use database transactions and/or optimistic concurrency for inventory correctness
- Add observability before load testing and cloud deployment

## Proposed Architecture

Initial architecture: modular monolith.

Backend solution structure:

```text
FlashSale.sln
src/
  FlashSale.Api/
  FlashSale.Application/
  FlashSale.Domain/
  FlashSale.Infrastructure/
  FlashSale.Worker/
tests/
  FlashSale.UnitTests/
  FlashSale.IntegrationTests/
```

Responsibilities:

- `FlashSale.Api`: HTTP endpoints, auth, middleware, OpenAPI, health checks
- `FlashSale.Application`: use cases, commands, queries, validation
- `FlashSale.Domain`: entities, domain rules, enums, value objects
- `FlashSale.Infrastructure`: PostgreSQL, Redis, queue adapters, external services
- `FlashSale.Worker`: background jobs such as reservation expiry and notifications
- `tests`: unit and integration tests, ideally using Testcontainers

Frontend structure:

```text
web/
  src/
    app/
    pages/
    components/
    features/
      auth/
      products/
      flash-sales/
      orders/
      admin/
    api/
    hooks/
    types/
```

## Purchase Flow

Initial target flow:

1. Customer opens active flash sale campaign.
2. Customer clicks buy.
3. API validates campaign is active.
4. API validates item is available.
5. API creates a short-lived reservation.
6. Customer confirms checkout/payment.
7. API creates order using an idempotency key.
8. Reservation becomes sold inventory.
9. Expired reservations are released by background worker.

- Add flash sale campaign CRUD

### Phase 2: Correct Flash Sale Behavior

Goal: handle concurrency safely.

- Add Redis
- Add inventory reservation with TTL
- Add idempotency key support
- Add rate limiting
- Add concurrency control for inventory updates
- Add background worker for expired reservations
- Add integration tests for concurrent purchase attempts

### Phase 3: User Experience and Realtime

Goal: make the sale experience feel live.

- Add countdown UI
- Add SignalR for live stock/sold-out updates
- Add order status updates
- Add loading, retry, empty, and error states
- Add admin dashboard metrics

### Phase 4: Observability and Hardening

Goal: prepare for serious testing and deployment.

- Add Serilog structured logging
- Add OpenTelemetry traces and metrics
- Add health checks
- Add security headers
- Add refresh tokens if needed
- Add audit logs for admin actions
- Add API versioning if needed
- Add load-test scripts with k6 or JMeter

### Phase 5: Docker Packaging

Goal: run the whole system locally with containers.

- Add Dockerfile for API
- Add Dockerfile for Worker
- Add Dockerfile for Web
- Add docker-compose.yml
- Add PostgreSQL container
- Add Redis container
- Add RabbitMQ or queue container if selected
- Add `.env.example`

### Phase 6: Scale Strategy

Goal: support high traffic.

- Keep API stateless
- Scale API instances horizontally
- Use Redis cache for hot sale/campaign data
- Use queue for non-critical async work
- Tune PostgreSQL indexes and connection pooling
- Add CDN for frontend/static assets
- Add load balancer
- Add autoscaling
- Add rate limiting and waiting-room strategy
- Benchmark hot paths

### Phase 7: Cloud Deployment

Azure option:

- Frontend: Azure Static Web Apps or Azure Storage Static Website + CDN
- Backend: Azure Container Apps or Azure App Service
- Database: Azure Database for PostgreSQL
- Cache: Azure Cache for Redis
- Queue: Azure Service Bus
- Registry: Azure Container Registry
- Monitoring: Application Insights and Azure Monitor
- Scale advanced option: AKS

AWS option:

- Frontend: S3 + CloudFront
- Backend: ECS Fargate or EKS
- Database: RDS PostgreSQL
- Cache: ElastiCache Redis
- Queue: SQS/SNS or Amazon MQ
- Registry: ECR
- Monitoring: CloudWatch and X-Ray
- Load balancer: ALB
