## What Was Completed Since Last Handoff

### Order stock update moved behind an abstraction

New API stock updater contract:

```text
src\FlashSale.Api\Orders\IFlashSaleStockUpdater.cs
```

Production implementation:

```text
src\FlashSale.Api\Orders\EfFlashSaleStockUpdater.cs
```

`EfFlashSaleStockUpdater` uses EF Core `ExecuteUpdateAsync` so PostgreSQL performs the stock increment only when the row still has enough available stock:

```text
SoldQuantity + requested quantity <= TotalQuantity
```

This makes the stock write safer than the previous application-only pattern where the API read `SoldQuantity`, checked it in memory, then later assigned a new value.

### Order endpoint updated

Updated:

```text
src\FlashSale.Api\Orders\OrderEndpoints.cs
```

`POST /api/orders` now receives:

```csharp
IFlashSaleStockUpdater stockUpdater
```

The endpoint still validates request shape, item existence, item active status, campaign active status, campaign time window, and available stock before beginning order creation.

Inside the transaction it now calls the stock updater for each distinct `FlashSaleItemId`. If any stock update fails, the transaction rolls back and the API returns:

```json
{
  "message": "Not enough stock."
}
```

The old direct mutation was removed from the order item creation loop:

```csharp
flashSaleItem.SoldQuantity += requestItem.Quantity;
flashSaleItem.UpdatedAt = now;
```

## Recommended Next Tasks

### Step 1: Add PostgreSQL-backed concurrency integration tests

Add tests that use real PostgreSQL instead of EF InMemory for concurrency behavior:

- Two concurrent requests compete for the final unit of stock.
- Duplicate lines for the same `FlashSaleItemId`.
- Multi-item order rolls back if one item has insufficient stock.
- Successful multi-item order increments stock for every item in the same transaction.

### Step 2: Improve stock failure messages later

The production updater currently returns `false` for inactive, missing, or insufficient-stock rows. The endpoint validates most of these cases before the update, so the common failure should be concurrent insufficient stock.

Future refinement can return richer failure reasons if needed.

### Step 3: Design idempotency keys

Before payment or checkout finalization, design idempotency keys for duplicate client submissions.

### Step 4: Keep reservation TTL separate

Do not add reservation TTL yet. The current rule remains: creating an order immediately increases `SoldQuantity`.
