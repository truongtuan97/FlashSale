## Scope

This rules file defines backend rules and flows for the FlashSale project. New backend implementation should follow this file unless a newer `rules-ddmmyy-HHMM` file supersedes it.

## Repository Hygiene Rules

- Do not commit build artifacts, runtime logs, local IDE files, test output, coverage output, or generated `bin`/`obj` files.
- `.gitignore` must ignore:
  - `**/[Bb]in/`
  - `**/[Oo]bj/`
  - `*.log`
  - `TestResults/`
  - coverage outputs
  - `.vs/`
  - local user files
- After every compile or test run, `git status --short` should only show real source, migration, test, rule, or handoff files.
- If `bin`, `obj`, or log files appear in git status, fix `.gitignore` or untrack those files before committing.

## Migration Rules

- Database schema changes must use EF Core migrations.
- Do not manually edit PostgreSQL tables for schema changes.
- Always inspect generated migrations before applying them.
- If EF generates destructive operations such as unrelated `DropTable`, `DropColumn`, or data-losing changes, stop and inspect before running `database update`.
- For column renames, prefer `RenameColumn` over `DropColumn` plus `AddColumn` to preserve data.
- After applying migrations, verify with:

```powershell
dotnet ef migrations list `
  --project src\FlashSale.Infrastructure `
  --startup-project src\FlashSale.Api
```

## Product Rules

- `Product` is the base catalog product.
- Product CRUD endpoint group:

```text
GET    /api/products
GET    /api/products/{id}
POST   /api/products
PUT    /api/products/{id}
DELETE /api/products/{id}
```

- Product name is required.
- Product price must be greater than `0`.
- Product response mapping should be covered by unit tests.

## Campaign Rules

- `FlashSaleCampaign` is the sale event, for example 7.7, 8.8, Black Friday.
- Campaign endpoint group:

```text
GET    /api/flash-sale-campaigns
GET    /api/flash-sale-campaigns/{id}
POST   /api/flash-sale-campaigns
PUT    /api/flash-sale-campaigns/{id}
DELETE /api/flash-sale-campaigns/{id}
```

- Campaign name is required.
- `EndsAt` must be later than `StartsAt`.
- Use `StartsAt` and `EndsAt`, not `StartAt` and `EndAt`.
- Campaign time rules must use `DateTimeOffset`.

## Flash Sale Item Rules

- `FlashSaleCampaign` must not directly own products.
- Correct relationship:

```text
FlashSaleCampaign 1 - N FlashSaleItem N - 1 Product
```

- `Product` stores catalog data.
- `FlashSaleCampaign` stores the sale event.
- `FlashSaleItem` connects a product to a campaign and stores sale-specific data such as sale price and stock.
- The same product may appear in multiple campaigns.
- The same product must not be added twice to the same campaign.
- Enforce unique index on `(CampaignId, ProductId)`.
- Money fields use precision `(18, 2)`.
- FK delete behavior from `FlashSaleItem` to `Product` and `FlashSaleCampaign` should be `Restrict`.

Flash sale item validation rules:

- `CampaignId` is required.
- `ProductId` is required.
- Campaign must exist before creating or updating item.
- Product must exist before creating or updating item.
- Duplicate `(CampaignId, ProductId)` returns conflict.
- `OriginalPrice > 0`.
- `SalePrice > 0`.
- `SalePrice <= OriginalPrice`.
- `TotalQuantity > 0`.
- `SoldQuantity >= 0`.
- `SoldQuantity <= TotalQuantity`.
- `AvailableQuantity = TotalQuantity - SoldQuantity`.

```text

```

## Order Domain Rules

- Users buy `FlashSaleItem`, not `Product` directly.
- `Order` is the customer-level purchase.
- `OrderItem` is one purchased flash sale item line inside an order.
- Relationship:

```text
Order 1 - N OrderItem
OrderItem N - 1 FlashSaleItem
FlashSaleItem N - 1 Product
FlashSaleItem N - 1 FlashSaleCampaign
```

- `Order.Status` starts as `Pending`.
- Valid initial statuses:
  - `Pending = 1`
  - `Paid = 2`
  - `Cancelled = 3`
  - `Failed = 4`
- `Order.TotalAmount` is the sum of all `OrderItem.LineTotal`.
- `OrderItem.LineTotal = Quantity * UnitPrice`.
- `OrderItem.UnitPrice` must be a snapshot of `FlashSaleItem.SalePrice` at order creation time.
- Existing orders must not change if product price or flash sale price changes later.
- `Order -> OrderItems` delete behavior may be cascade.
- `OrderItem -> FlashSaleItem` delete behavior must be restrict to preserve order history.

## Create Order Flow

The `POST /api/orders` endpoint must follow this order:

1. Validate `CustomerName` and `CustomerEmail`.
2. Validate `Items` is not empty.
3. Validate every item has `Quantity > 0`.
4. Load requested `FlashSaleItems` with `Campaign` and `Product`.
5. Check every requested `FlashSaleItem` exists.
6. Check every requested `FlashSaleItem` is active.
7. Check every related `FlashSaleCampaign` is active.
8. Check current UTC time is within `Campaign.StartsAt` and `Campaign.EndsAt`.
9. Check stock with `SoldQuantity + requested Quantity <= TotalQuantity`.
10. Create `Order`.
11. Create `OrderItems`.
12. Snapshot `OrderItem.UnitPrice = FlashSaleItem.SalePrice`.
13. Calculate `OrderItem.LineTotal = Quantity * UnitPrice`.
14. Calculate `Order.TotalAmount = sum(OrderItem.LineTotal)`.
15. Increase `FlashSaleItem.SoldQuantity` by requested quantity.
16. Save changes inside a database transaction.

## Business Rules

- The initial implementation increases `SoldQuantity` immediately when the order is created.
- Overselling is not allowed.
- Campaign time checks must use UTC.
- Stock check must be:

```text
FlashSaleItem.SoldQuantity + requested Quantity <= FlashSaleItem.TotalQuantity
```

- If the same `FlashSaleItemId` appears more than once in a create-order request, stock checks must use the aggregated requested quantity for that flash sale item.
- If order creation succeeds, increase `FlashSaleItem.SoldQuantity` in the same transaction.
- If order creation fails, do not change `SoldQuantity`.
- Save order creation and stock updates inside a database transaction.
- Current implementation does not yet use reservation TTL or idempotency keys.
- Do not rush into payment/checkout finalization until concurrency and idempotency are designed.

## Expected Failure Responses

Use `400 Bad Request` for validation and business-rule failures:

- Customer name is required.
- Customer email is required.
- Order must contain at least one item.
- Flash sale item is required.
- Quantity must be greater than 0.
- Flash sale item does not exist.
- Flash sale item is not active.
- Campaign is not active.
- Campaign is not currently running.
- Not enough stock.

## Current Order Endpoint Implementation Notes

- Primary endpoint class currently exists in:

```text
src/FlashSale.Api/Orders/OrderEndpoints.cs
```

- Endpoint group is `/api/orders`.
- `POST /api/orders` creates `Order` and `OrderItems`, snapshots unit price, calculates line totals and total amount, increases sold quantity, and returns `201 Created`.
- Endpoint naming should stay consistent:
  - `OrderEndpoints`
  - `MapOrderEndpoints`
- Create and read order responses should include order item details.
- `GET /api/orders/{id}` returns the order header plus item details. Each item includes `FlashSaleItemId`, `ProductName`, `CampaignName`, `Quantity`, snapshot `UnitPrice`, and `LineTotal`.

## Testing Rules

Add unit tests for request validation before or alongside endpoint implementation.

Important integration tests:

- Can create order for an active campaign item.
- Rejects missing customer name.
- Rejects empty items.
- Rejects quantity <= 0.
- Rejects missing flash sale item.
- Rejects inactive flash sale item.
- Rejects inactive campaign.
- Rejects campaign outside current time window.
- Rejects oversell.
- Created order returns correct total amount and order item line totals.
- Creating order increases `FlashSaleItem.SoldQuantity`.
