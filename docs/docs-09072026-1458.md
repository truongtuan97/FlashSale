# PostgreSQL Order Concurrency Coverage

Created at: 2026-07-09 14:58 Asia/Bangkok

## Context

Order stock updates now go through `IFlashSaleStockUpdater`.

The production implementation, `EfFlashSaleStockUpdater`, uses EF Core `ExecuteUpdateAsync` with a database-side stock guard:

```text
SoldQuantity + requested quantity <= TotalQuantity
```

The normal API integration tests still use EF Core InMemory and a test-only `InMemoryFlashSaleStockUpdater`. That keeps the broad API test suite fast and compatible with the InMemory provider, but it cannot prove PostgreSQL relational update behavior.

## Decision

Add a separate PostgreSQL-backed integration test path for order stock concurrency behavior.

The PostgreSQL tests use a dedicated `PostgresFlashSaleApiFactory` that:

- Replaces the API DbContext with Npgsql.
- Keeps the production `EfFlashSaleStockUpdater`.
- Creates a unique test database per factory instance.
- Applies EF Core migrations before running the test.
- Drops the test database during cleanup.

The PostgreSQL connection string is read from:

```text
FLASHSALE_TEST_POSTGRES
```

This keeps local credentials out of source control.

## Covered Scenarios

The PostgreSQL order concurrency tests cover:

- Two concurrent orders competing for the final unit of stock.
- Duplicate request lines for the same `FlashSaleItemId`.
- Multi-item order rollback when one item has insufficient stock.
- Successful multi-item order increments stock for every item.

## Non-Goals

This work does not add:

- Idempotency keys.
- Reservation TTL.
- Payment or checkout finalization.
- New database schema changes.

Those should remain separate design steps.
