# Order Concurrency Design

Created at: 2026-07-02 15:26 Asia/Bangkok

## Goal

Prevent overselling when many customers create orders for the same flash sale item at the same time.

The current `POST /api/orders` flow checks stock in application code, then increments `SoldQuantity` before saving. This is fine for early behavior tests, but it is not enough for production flash sale traffic because two requests can read the same available stock before either one commits.

## Recommended First Production Step

Use a conditional stock update in PostgreSQL as the first concurrency-safe implementation:

```sql
UPDATE "FlashSaleItems"
SET "SoldQuantity" = "SoldQuantity" + @quantity,
    "UpdatedAt" = @now
WHERE "Id" = @flashSaleItemId
  AND "IsActive" = TRUE
  AND "SoldQuantity" + @quantity <= "TotalQuantity";
```

Then check affected rows:

- `1` affected row means stock was reserved successfully.
- `0` affected rows means the item is inactive, missing, or does not have enough stock after concurrent updates.

The API should still load the item with campaign and product before creating the order so it can validate campaign status, campaign time window, and snapshot product/campaign details for the response.

## Multi-Item Orders

For orders with multiple distinct flash sale items, aggregate duplicate `FlashSaleItemId` values before updating stock.

Run all conditional updates inside the same database transaction as order creation:

1. Validate request shape.
2. Aggregate requested quantity by `FlashSaleItemId`.
3. Load all requested flash sale items with campaign and product.
4. Validate item existence, item active status, campaign active status, and campaign time window.
5. Run one conditional stock update per distinct flash sale item.
6. If any update affects `0` rows, roll back and return `400 Bad Request` with `Not enough stock.` or a more specific future error.
7. Create `Order` and `OrderItems`.
8. Commit the transaction.

This keeps order rows and stock changes atomic.

## Why Conditional Update First

Conditional update is the best next step because it is simple, database-enforced, and does not require a new schema column.

Compared with application-only checks, it prevents the classic race where two requests both see enough stock and both save.

Compared with explicit `SELECT ... FOR UPDATE`, it keeps the stock decision close to the write itself and reduces the amount of lock-management code in the API.

Compared with optimistic concurrency tokens, it is easier to introduce before the domain needs general row-version conflict handling.

## Future Design Work

Idempotency should be added before payment or checkout finalization. A future `IdempotencyKey` should prevent duplicate orders from repeated client submissions.

Reservation TTL should be designed later if the system separates stock reservation from payment completion. Until then, the current rule remains: creating an order immediately increases `SoldQuantity`.

## Testing Plan

The current integration tests use EF Core InMemory, which cannot prove PostgreSQL concurrency behavior.

When conditional stock update is implemented, add PostgreSQL-backed integration tests for:

- Two concurrent requests competing for the final unit of stock.
- Duplicate lines for the same `FlashSaleItemId`.
- Multi-item order rollback when one item has insufficient stock.
- Successful order creates order rows and increments all requested stock in one transaction.

Keep the existing InMemory tests for general API behavior, validation responses, and response shape.
